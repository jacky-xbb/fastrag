# 指标表格按「指标行」切块 + 向量/BM25 混合检索 + 元数据过滤

知识库的核心检索对象是**指标**（见 [CONTEXT.md](../../CONTEXT.md)）——表格里的量化数字（拉力、撕裂强度、吸水率…）。围绕这点定三条，**不是可选优化，是「指标问答答得准」的前提**：

1. **指标表格不整块嵌入，按「指标行」切块**。每块前缀加上语义锚点：`标准号 + 表名 + 指标名`（如 `GB/T 23457-2017 表2 物理力学性能 / 拉力`），让裸数字带着上下文进向量空间。
2. **检索用向量 + 关键词（BM25/FTS）混合**，不是纯向量。
3. **带元数据过滤**：每个 chunk 存 `{标准号, 表名, 指标名, 页码, 状态(现行/废止)}`，指标问答优先靠精确匹配收窄。

理由：

- 稠密向量**天生对精确数字 / 表格格子查找弱**，「800 对应 PY 那一列」这种语义嵌入抓不住。这是嵌入式检索的固有短板，**换更大的 embedding（3-large）也救不了**——所以 embedding 锁 `text-embedding-3-small`（见调研文档 §11），别在模型大小上浪费精力。
- OCR（PaddleOCR-VL，见 [ADR-0003](0003-ocr-paddleocr-vl.md)）输出的是**大块 HTML 表格**。整张表塞进一个 chunk 会把向量冲淡，反而更难命中——必须先拆。

代价：切块要解析 HTML 表、重建行上下文，比直接上 Mastra 的 `recursive` 策略多写代码；混合检索要在 libSQL 向量之外再加一层 BM25/全文索引。认这个代价。

## VL 表格的实际格式（已实测 4 份，写 chunker 前先看这条）

PaddleOCR-VL 的表格输出**不是 markdown 管道表，是 HTML `<table>`**，且格式跨文档一致（实测 GB/T 23457、jc684-1997、GB/T 18242、JC/T 974，含多层嵌套表）：

- **`<tr>/<td>` 带 `rowspan`/`colspan`**，合并单元格原样保留——切块要顺着 span 还原「这个数字属于哪行哪列」，不能按纯文本切。
- **单位 / 公式 / 上标用 LaTeX 包着**：`$g/m^{2}$`、脚注 `$^{a}$`、`$4 mm^{2}$`。chunker 要把这些归一化成纯文本（如 `g/m²`），否则 `$...$` 噪声会进 chunk 和向量。
- **个别单元格内有转义 `\n`**（如 `0.3 MPa,\n120 min\n不透水`），要当空格/换行清掉。

→ chunker 第一步是「HTML 表解析 + LaTeX 单位归一化 + `\n` 清洗」，再按指标行切。
